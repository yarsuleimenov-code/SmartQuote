Reference Architecture v1.1

Страница показывает полный lifecycle от draft quote до completed delivery. Используется для согласования границ между commercial workflow, pricing engine и operational execution.

Core Rule:

Draft editable, Estimate immutable

Назначение lifecycle map

Lifecycle map фиксирует, какие сущности создаются на каждом этапе, какие данные можно редактировать, а какие должны становиться immutable snapshot.

Это защищает систему от смешивания quote, estimate, order и eBOL в одну неуправляемую сущность.

Главная задача страницы — показать, где заканчивается commercial layer и где начинается operational layer.

Pricing не должен изменяться после создания estimate snapshot. Orders и eBOL работают только с зафиксированными данными.

Lifecycle Stages

Ownership, mutability и назначение каждого этапа.

Draft

Mutable

Draft — рабочее состояние quote, которое брокер может редактировать и пересчитывать.

Owner

Commercial Layer / Broker

Allowed Actions

Edit items, route, access conditions, quote options, recalculate.

Estimate

Immutable

Estimate — customer-facing snapshot цены, созданный из Draft после Generate Estimate.

Owner

Commercial Layer / Snapshot Service

Allowed Actions

Send, view, approve, reject, reopen as new draft.

Invoice

Billing

Invoice создаётся на основе approved estimate и не должен пересчитывать pricing.

Owner

Commercial Layer / Billing

Allowed Actions

Send invoice, track payment, mark paid, cancel.

Order

Operational

Order — operational entity для выполнения перевозки после commercial approval.

Owner

Operational Layer / Dispatch

Allowed Actions

Schedule, assign route, assign crew, update operational statuses.

eBOL

Proof

eBOL фиксирует item verification, pickup/delivery statuses, photos, signatures и exceptions.

Owner

Operational Layer / Field Team

Allowed Actions

Verify items, upload photos, collect signature, report exceptions.

Completed Delivery

Closed

Completed state означает, что delivery подтверждена, POD создан, а operational workflow закрыт.

Owner

Operational Layer / Manager

Allowed Actions

View only, audit, export documents, review exceptions.

Lifecycle Risks

Главные ошибки, которые должны быть предотвращены архитектурой.

Critical

Risk 1 — Direct Estimate Editing

Если уже отправленный estimate можно редактировать напрямую, historical pricing становится недостоверным.

Решение: любые изменения создают new draft и new estimate version.

Risk 2 — Order Changes Pricing

Operational workflow не должен менять commercial pricing после approval.

Решение: Order работает только с frozen estimate snapshot.

Risk 3 — eBOL Without Snapshot

eBOL без item snapshot может потерять связь с изначальным заказом.

Решение: eBOL создаётся из Order и frozen item list.

Risk 4 — Lifecycle Overload

Если broker screen начнёт включать dispatch/eBOL логику, quote creation станет медленнее.

Решение: Calculator = quote creation, eBOL = operational execution.

1. Draft vs Estimate

Draft является рабочим состоянием quote. Estimate является frozen customer-facing snapshot.

Это одно из главных архитектурных разделений системы.

2. Snapshot Integrity

Historical estimates не должны пересчитываться после:

  • изменения fuel variables;
  • изменения formulas;
  • изменения pricing logic;
  • изменения route rules.

3. Commercial Layer vs Operational Layer

Calculator отвечает за:

  • pricing;
  • quote generation;
  • estimate lifecycle;
  • commercial workflow.

Orders и eBOL отвечают за:

  • dispatch;
  • pickup;
  • delivery;
  • POD;
  • exceptions.

4. Main Lifecycle Rule

Draft → Editable
Estimate → Immutable
Order → Operational
eBOL → Verification & POD

5. Long-Term Architecture Goal

В production architecture:

  • pricing logic должна жить в backend pricing engine;
  • frontend не должен содержать hidden pricing rules;
  • all lifecycle transitions должны быть auditable;
  • every estimate должен быть reproducible.
